For **20 years**, my job came down to one image: a screen full of tags, braces and angle brackets. Code. `<div>`, `<Component />`, `<T>`. Line after line, I wrote the translation of my ideas into instructions a machine could run.

Today the image has changed. The cursor still blinks, but what I type looks more like a **conversation** than a syntax. `> Add pagination`. `> Refactor this service`. `> Explain this bug`. The `<.../>` has become a `>/...`.

This post is about that **shift in the craft**: from the artisan who writes the code to the **conductor** who has it written.

## 1. The `<.../>` era: writing every line

When I started, a developer's worth was measured by the ability to **produce correct code**. Knowing the syntax, memorizing APIs, taming frameworks, debugging by hand. The skill lived in the **fingers** as much as in the head.

We spent hours:

- hunting for the **right signature** of a method in the docs
- retyping **boilerplate** we'd already written a hundred times
- chasing a missing semicolon or a typo
- reinventing known **patterns** because it was faster than finding them again

Code was both the **means** and the **proof** of the work. A productive day meant hundreds of lines committed.

## 2. The tipping point: from keyboard to prompt

The arrival of **AI agents** able to understand intent and generate coherent code shifted the craft's center of gravity.

I no longer write the `for` loop. I **describe what it must do**. I no longer look up the method signature. I **ask**, and the agent knows it. Boilerplate? Generated in one sentence.

The keyboard still matters, but for something else: to **phrase**, **frame**, **correct**. The skill has migrated from the fingers to the **statement of the problem**.

> Before, I knew *how* to write the code.  
> Now, I mostly need to know *what* to ask for and *why*.

## 3. What actually changes in the job

The trap is thinking the job gets **easier**. It mostly gets **different**.

**What loses value:**

- memorizing syntax
- typing speed
- knowing APIs by heart

**What gains a lot:**

- **clarity of intent** — a vague prompt produces vague code
- **the ability to judge** — the AI proposes, I must know if it's correct, safe, maintainable
- **architecture** — deciding *what* to build stays deeply human
- **code review** — reading becomes a core skill, not a chore

The experience built over 20 years doesn't vanish: it gets **reinvested**. Spotting a bad abstraction, sensing a security hole, anticipating technical debt — that's exactly what lets me **steer** the agent instead of being steered by it.

## 4. The new role: conductor, not performer

I now see myself less as a **musician** playing each note, and more as a **conductor**. The AI agent is a brilliant instrumentalist — fast, tireless — but without direction it plays out of tune.

My work is to:

1. **Break down** the problem into clear intents
2. **Delegate** the generation to the agent
3. **Verify** without indulgence what comes back
4. **Iterate** until the result holds up

Code is still the deliverable. But it's no longer written **by** me. It's written **through** me.

## Conclusion: the bracket changed meaning

In 20 years, I moved to the other side of the bracket. The `<.../>` I used to **write** has become the `>/...` I now **dictate**.

The job didn't disappear — it **moved up a floor**. For a long time we were paid to *type* code. More and more, we're paid to **know which code deserves to exist**, and to judge the code a machine proposes.

> Syntax was our mother tongue.  
> Intent is becoming our new language.

And somehow that's reassuring: the machine writes the lines, but it's still the human who decides the **story** they tell.
